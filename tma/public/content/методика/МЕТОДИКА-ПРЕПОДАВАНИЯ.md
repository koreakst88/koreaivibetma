## 1. Как вести урок

### Общие принципы

**1.1. Структура каждого занятия**

```
┌─────────────────────────────────────────────────────────┐
│  СТАНДАРТНАЯ СТРУКТУРА УРОКА (60 минут)                │
│                                                         │
│  0-5 мин   │ Приветствие + проверка ДЗ                  │
│  5-20 мин  │ Теория + демонстрация                      │
│  20-50 мин │ Практика (ученик делает, вы наблюдаете)    │
│  50-55 мин │ Проверка результата + исправление ошибок   │
│  55-60 мин │ Домашнее задание + ответы на вопросы       │
└─────────────────────────────────────────────────────────┘
```

---

**1.2. Подготовка к уроку**

**За 30 минут до начала:**

- [ ] Проверить связь (Zoom/платформа)
- [ ] Открыть все необходимые файлы (День X, шпаргалки)
- [ ] Открыть ChatGPT/Claude в браузере
- [ ] Проверить, что Cline работает в VS Code
- [ ] Подготовить демонстрационный проект
- [ ] Включить запись урока (если нужно)

**Чек-лист готовности:**
- [ ] Интернет стабилен
- [ ] Микрофон работает
- [ ] Камера работает
- [ ] Демонстрация экрана настроена
- [ ] Все ссылки под рукой

---

**1.3. Начало урока**

**Первые 5 минут критичны для настроения ученика:**

```
✅ ЧТО ДЕЛАТЬ:
- Приветствовать по имени
- Спросить "Как дела?" (искренне)
- Спросить про домашнее задание
- Похвалить за прогресс ("Видел, вы сделали...")
- Создать позитивную атмосферу

❌ ЧЕГО ИЗБЕГАТЬ:
- Сразу переходить к материалу
- Игнорировать вопросы "как дела"
- Критиковать за невыполненное ДЗ
- Спешить
```

**Пример начала:**
> "Привет, [Имя]! Рад видеть! Как настроение?
> Как прошло домашнее задание? Что получилось?
> [Выслушать, похвалить]
> Отлично! Сегодня у нас [тема дня] — будет интересно!
> Начнём?"

---

**1.4. Подача материала**

**Правило 10/20/30:**
- **10 минут** — теория (не больше!)
- **20 минут** — демонстрация (вы показываете)
- **30 минут** — практика (ученик делает)

**Принципы подачи:**

| Принцип | Как применять |
|---|---|
| **Простыми словами** | Избегать терминов "API", "SDK", "webhook" без объяснения |
| **На примерах** | "Представьте, что API — это официант в ресторане..." |
| **Визуализация** | Рисовать схемы, показывать на экране |
| **Повторение** | После каждого блока: "Давайте повторим, что прошли" |
| **Проверка понимания** | "Всё понятно? Есть вопросы?" (пауза 5 секунд) |

---

**1.5. Демонстрация**

**Когда показываете на своём экране:**

```
✅ ПРАВИЛА:
- Говорить вслух, что делаете ("Сейчас я нажимаю...")
- Объяснять ЗАЧЕМ ("Это нужно, чтобы...")
- Делать паузы ("Успели записать?")
- Показывать типичные ошибки ("Если увидите это...")

❌ ОШИБКИ:
- Молча делать
- Говорить слишком быстро
- Пропускать шаги ("Тут всё просто...")
- Показывать только идеальный результат
```

---

**1.6. Практика ученика**

**Когда ученик делает сам:**

```
✅ ЧТО ДЕЛАТЬ:
- Наблюдать (не вмешиваться сразу)
- Давать подсказки, а не готовые ответы
- Хвалить за прогресс ("Отлично!")
- Записывать ошибки (обсудить потом)

❌ ЧЕГО НЕ ДЕЛАТЬ:
- Не хватать мышку/клавиатуру
- Не делать ЗА ученика
- Не критиковать ("Ну как же так...")
- Не торопить ("Быстрее, быстрее...")
```

**Если ученик медленно:**
> "Всё нормально, я тоже сначала медленно делал.
> Главное — понять, а скорость придёт.
> Продолжайте, я рядом."

---

**1.7. Завершение урока**

**Последние 5 минут:**

```
1. Подвести итоги ("Сегодня мы сделали...")
2. Похвалить ("Вы молодец, особенно...")
3. Объяснить ДЗ (чётко, письменно в чат)
4. Спросить про вопросы ("Что осталось непонятно?")
5. Попрощаться позитивно ("До встречи!")
```

**После урока:**
- [ ] Отправить запись (если записывали)
- [ ] Отправить конспект/шпаргалки
- [ ] Напомнить про ДЗ
- [ ] Записать заметки по ученику (для себя)

## 2. Типичные вопросы учеников и ответы

### Технические вопросы

**В: "А зачем нам Git? Можно же просто файлы отправлять"**

**Ответ:**
> "Отличный вопрос! Git — это как 'сохранения' в игре.
> Если что-то сломается — можно откатиться назад.
> Плюс: GitHub хранит ваши проекты бесплатно и надёжно.
> Без Git: удалили файл случайно — всё, пропал.
> С Git: одна команда — и файл вернулся.
> Стоит потратить 10 минут на изучение!"

---

**В: "Почему именно Vercel? А другие хостинги?"**

**Ответ:**
> "Vercel идеален для начала:
> - Бесплатно для наших проектов
> - Деплой за 2 клика (подключил GitHub — готово)
> - Автоматические обновления (push → сайт обновился)
> - HTTPS автоматически
> 
> Другие варианты: Netlify (аналогично), GitHub Pages (чуть сложнее).
> Когда освоитесь — сможете попробовать любое."

---

**В: "Local Storage — это надёжно? Данные не пропадут?"**

**Ответ:**
> "Для MVP — да, надёжно.
> LocalStorage хранится в браузере пользователя.
> 
> Пропадёт, если:
> - Пользователь почистит кэш браузера
> - Откроет с другого устройства
> 
> Для полноценного проекта нужна база данных (Supabase, Firebase).
> Но для старта и тестирования — LocalStorage отлично!
> 90% MVP начинают с него."

---

**В: "А если Telegram изменит API? Всё переделывать?"**

**Ответ:**
> "Хороший вопрос! Telegram меняет API редко.
> Но если изменится — ИИ поможет обновить код.
> 
> Главное: вы научились ПОДХОДУ, а не конкретному коду.
> Код можно перегенерировать за минуты.
> Понимание архитектуры — остаётся навсегда."

---

**В: "Cline платный? Сколько стоит?"**

**Ответ:**
> "Cline — бесплатное расширение.
> Но нужен API ключ (ChatGPT или Claude).
> 
> ChatGPT: бесплатно (GPT-3.5) или $20/мес (GPT-4)
> Claude: бесплатно (лимит ~30 сообщений/день)
> 
> Для старта хватит бесплатных версий!
> Когда начнёте зарабатывать — обновите."

---

## 3. Что делать, если ученик застрял

### Распознавание проблемы

**Признаки того, что ученик застрял:**

| Признак | Что означает |
|---|---|
| Долго молчит | Не понимает, но стесняется сказать |
| "Эээ...", "Ну..." | Пытается сформулировать вопрос |
| "А как это..." | Конкретный вопрос, но не знает как спросить |
| Смотрит в экран без действий | Не знает, что делать дальше |
| Делает ошибки снова и снова | Не понял принцип |

---

### Алгоритм помощи

**Шаг 1: Выяснить проблему**

```
❓ Вопросы:
- "На каком шаге застряли?"
- "Что именно не получается?"
- "Какую ошибку видите?"
- "Что ожидали, а что получилось?"

✅ Слушать активно, не перебивать
```

---

**Шаг 2: Локализовать**

```
Разбить проблему на части:
- Это техническая ошибка (код не работает)?
- Это непонимание (не знает, что делать)?
- Это страх (боится нажать не туда)?

Пример:
"Давайте проверим: код написан, но не работает?
Или не знаете, какой код написать?"
```

---

**Шаг 3: Помочь (но не делать за ученика)**

```
✅ ПРАВИЛЬНО:
- Дать подсказку: "Посмотри в шпаргалку, раздел..."
- Показать НА СВОЁМ экране: "Смотри, я делаю так..."
- Объяснить принцип: "Здесь логика такая..."
- Дать сделать САМОМУ: "Теперь попробуйте вы"

❌ НЕПРАВИЛЬНО:
- Сделать за ученика
- Сказать "это просто" (обесценивает)
- Торопить: "Да быстрее же!"
- Критиковать: "Ну как же так..."
```

---

**Шаг 4: Проверить понимание**

```
После помощи:
- "Теперь понятно?"
- "Попробуйте объяснить своими словами"
- "Сделайте ещё раз самостоятельно"

Если понял — идём дальше.
Если нет — повторить с другой формулировкой.
```

---

### Частые проблемы и решения

**Проблема 1: "Ничего не работает!"**

**Диагноз:** Обычно что-то конкретное, но ученик обобщает.

**Решение:**
```
1. Попросить показать экран
2. Открыть консоль (F12)
3. Найти конкретную ошибку
4. Исправить вместе
5. Объяснить: "Вот эта строка вызывала ошибку, потому что..."
```

---

**Проблема 2: "Я не понимаю, как управлять ИИ"**

**Диагноз:** Не усвоил формулу промптов.

**Решение:**
```
1. Вернуться к шпаргалке "Формула промптов"
2. Разобрать пример вместе
3. Написать промпт вместе с учеником
4. Отправить, посмотреть результат
5. Обсудить: "Видите, ИИ понял, потому что вы чётко описали"
```

---

**Проблема 3: "Слишком сложно, я не справлюсь"**

**Диагноз:** Потеря мотивации, страх.

**Решение:**
```
1. Признать чувства: "Понимаю, сначала кажется сложным"
2. Напомнить о прогрессе: "Но вы уже сделали..."
3. Разбить на мельчайшие шаги: "Сейчас только это..."
4. Поддержать: "Я рядом, вместе справимся"
5. После успеха: "Видите! Вы смогли!"
```

---

**Проблема 4: "Долго делаю, все быстрее"**

**Диагноз:** Сравнение с другими, неуверенность.

**Решение:**
```
1. Нормализовать: "Все учатся в своём темпе"
2. Привести пример: "Прошлый ученик тоже медленно начинал"
3. Сместить фокус: "Главное — понять, а не скорость"
4. Похвалить: "Зато вы делаете осознанно, это правильно"
```

---

**Проблема 5: "А зачем мне это?"**

**Диагноз:** Потеря мотивации, не видит ценности.

**Решение:**
```
1. Спросить: "Зачем вы пришли на курс?"
2. Напомнить о цели: "Вы хотели..."
3. Показать прогресс: "Уже есть приложение, которое..."
4. Нарисовать перспективу: "Через 3 дня будет..."
5. Вдохновить: "Это востребованный навык, вы сможете..."
```

## 5. Формат обратной связи

### Во время урока

**Позитивная обратная связь:**
```
✅ Конкретная похвала:
- "Отлично! Счётчик корзины работает — вы всё правильно сделали"
- "Видите? Вы сами справились! Молодец!"
- "Этот промпт написали идеально — ИИ понял с первого раза"

✅ Похвала усилия:
- "Вижу, вы старались — результат отличный"
- "Вы продвинулись далеко с прошлого раза"

✅ Похвала прогресса:
- "Неделю назад вы не знали терминал, а сейчас..."
- "Сравните: первое приложение и текущее — небо и земля"
```

**Корректирующая обратная связь:**
```
✅ Через "и" вместо "но":
- "Хорошо, И теперь давай исправим эту ошибку"
- (Не: "Хорошо, НО есть ошибка")

✅ Конструктивно:
- "Здесь можно сделать так..."
- "Попробуйте добавить..."

✅ С поддержкой:
- "Это частая ошибка, все через это проходят"
- "Сейчас разберёмся, вместе справимся"
```

## 6. Адаптация под разные уровни

### Уровни учеников

| Уровень | Характеристики | Подход |
|---|---|---|
| **Полный новичок** | Никогда не программировал, боится терминала | Медленнее, больше поддержки, простые аналогии |
| **Продвинутый новичок** | Знает основы, пробовал конструкторы | Стандартный темп, меньше объяснений базового |
| **Опытный** | Знает HTML/CSS, хочет добавить ИИ | Быстрее, акцент на ИИ-инструментах, сложные задачи |

---

### Адаптация материала

**Для новичка:**
```
- Больше аналогий ("API как официант...")
- Больше повторений
- Меньше терминов
- Чаще проверять понимание
- Больше похвалы
```

**Для опытного:**
```
- Меньше базовых объяснений
- Больше сложных задач
- Глубже в технические детали
- Быстрее темп
- Больше самостоятельной работы
```

---

### Индивидуальный подход

**Визуал:**
- Показывать на экране
- Рисовать схемы
- Давать скриншоты

**Аудиал:**
- Объяснять вслух
- Обсуждать каждый шаг
- Давать аудио-конспекты

**Кинестетик:**
- Давать делать самому
- Минимум теории, максимум практики
- "Попробуйте, почувствуйте"
