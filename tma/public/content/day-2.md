## Шаг 1: Что такое Telegram Mini Apps + Структура проекта (25 мин)

### Часть 1: Простыми словами — что это?

Mini App (мини-приложение) — это веб-страница, которая открывается внутри Telegram. Не в браузере, а прямо в мессенджере.

**Представьте:**

Вы открываете обычный сайт в браузере — например, интернет-магазин. Вы видите:
- Каталог товаров
- Корзину
- Форму заказа

Это обычный веб-сайт.

**Теперь представьте то же самое, но внутри Telegram:**

1. Вы открываете бота в Telegram
2. Нажимаете кнопку «Открыть магазин»
3. Внутри Telegram открывается приложение — каталог, корзина, форма
4. Вы выбираете товары, оформляете заказ
5. Всё это не выходя из Telegram!

Это и есть Mini App.

**В чём фишка?**

- ✅ Пользователь не уходит из Telegram
- ✅ Удобно — всё в одном месте
- ✅ Быстро — не нужно устанавливать отдельное приложение
- ✅ Доступ к данным пользователя Telegram (имя, ID, фото)
- ✅ Можно принимать платежи прямо в Telegram

Наше приложение доставки пиццы будет именно таким. Пользователь откроет бота → увидит каталог пицц → закажет → всё внутри Telegram.

### Часть 2: Чем отличается от обычного бота?

Важно понять разницу:

| Обычный Telegram бот | Telegram Mini App |
|---|---|
| Чат с текстовыми сообщениями и кнопками | Полноценная веб-страница внутри Telegram |
| Команды: /start, /help | Красивый визуальный интерфейс (как сайт) |
| Бот отвечает текстом | Можно делать всё: формы, каталоги, корзины, анимации |
| Ограниченный интерфейс (только кнопки под сообщениями) | Выглядит профессионально |

**Примеры обычных ботов:** бот-переводчик, бот для поиска музыки.

**Примеры Mini Apps:** интернет-магазин, игра, сервис заказа еды.

**Можно комбинировать:**

Бот + Mini App = идеальная связка.

- Бот — для уведомлений и простых команд
- Mini App — для сложного интерфейса (каталог, заказ)

Это то, что мы и делаем!

### Часть 4: Как это технически работает?

Не будем грузить вас сложными терминами, но базовое понимание нужно.

**Простая схема:**

1. **Вы создаёте веб-страницу (HTML, CSS, JavaScript)**
   - Это обычный сайт, как мы делали вчера

2. **Вы публикуете её в интернет (деплой)**
   - Получаете ссылку: `https://ваше-приложение.vercel.app`

3. **Вы подключаете эту ссылку к Telegram боту**
   - Через @BotFather добавляете Mini App
   - Указываете ссылку на ваш сайт

4. **Пользователь открывает бота → видит кнопку → кликает → Mini App открывается**
   - Telegram загружает вашу веб-страницу внутри мессенджера

**Что делает страницу «Mini App»?**

Специальный SDK от Telegram:
- Подключаете скрипт `telegram-web-app.js`
- Теперь ваша страница «знает», что открыта в Telegram
- Получает доступ к данным пользователя (ID, имя)
- Может использовать кнопки Telegram (MainButton, BackButton)
- Адаптируется под тему (тёмная/светлая)

Звучит сложно, но на практике — просто:
1. Создаём обычную веб-страницу
2. Добавляем одну строчку: подключение SDK
3. Публикуем
4. Подключаем к боту
5. Готово!

ИИ сделает всё за вас. Вы просто даёте команды.

### Часть 5: Зачем нужна правильная структура?

Прежде чем начать писать код, нужно понять, КАК организовать файлы.

**Почему это важно?**

| Плохая структура | Хорошая структура |
|---|---|
| Все файлы в одной куче | Всё на своих местах |
| Непонятно, где что лежит | Легко ориентироваться |
| Сложно найти нужный файл | Можно масштабировать проект |
| Невозможно показать другому разработчику | Профессионально выглядит |

Мы сразу научимся делать правильно. Это как организация вещей в шкафу: носки отдельно, рубашки отдельно. Не нужно всё в одну кучу.

### Часть 6: Файловая структура проекта

Вот как будет выглядеть наш проект доставки пиццы:

```
/pizza-delivery
  /src
    - index.html       (главная страница — каталог)
    - cart.html        (страница корзины)
    - order.html       (страница оформления заказа)
    - style.css        (все стили)
    - script.js        (вся логика)
  /images
    - pizza1.jpg
    - pizza2.jpg
    - logo.png
  - README.md          (описание проекта)
  - .gitignore         (что не загружать на GitHub)
```

Разберём каждую часть.

#### /src — основная папка с кодом

Здесь лежат все файлы, которые видит пользователь.

- **index.html** — главная страница (каталог пицц)
  - Это то, что открывается первым
  - Здесь каталог из 6 пицц

- **cart.html** — страница корзины
  - Открывается при клике на «Корзина»
  - Показывает выбранные товары

- **order.html** — страница оформления заказа
  - Форма с полями: имя, телефон, адрес
  - Кнопка «Оформить заказ»

- **style.css** — все стили
  - Один файл со всеми CSS-правилами
  - Цвета, размеры, шрифты

- **script.js** — вся логика
  - Один файл со всем JavaScript
  - Добавление в корзину, расчёт суммы, валидация

**Почему всё в одном файле (CSS и JS)?**

Потому что проект небольшой. Для MVP проще иметь один файл — легче управлять.

#### /images — папка с картинками

Все изображения храним отдельно:
- Фотографии пицц
- Логотип
- Иконки

Так проще менять картинки, не трогая код.

Это текстовый файл с описанием проекта.

**Зачем?**
- Вы вспомните через месяц, что это за проект

### Часть 7: Документация — сохранение промптов

Теперь самое важное — сохранение промптов.

**Почему это критично?**

**Проблема без сохранения:**
Вы попросили ИИ создать функцию. Через неделю хотите добавить похожую. Не помните, как формулировали запрос. Приходится придумывать заново.

**Решение — файл с промптами:**

Создаём файл `prompts.md`. Туда записываем ВСЕ промпты, которые использовали.

**Пример содержания prompts.md:**

```markdown
# Промпты для проекта "Доставка пиццы"

## Создание каталога пицц
Создай каталог из 6 пицц в две колонки.
Каждая пицца: изображение (placeholder 300x300), название,
описание, цена, кнопка "Добавить в корзину".
Дизайн: красно-белая схема, современный стиль.

## Добавление корзины
Добавь функционал корзины: при клике на "Добавить в корзину"
товар сохраняется в массив. Покажи кнопку "Корзина" внизу
с количеством товаров.

## Форма заказа
Создай форму с полями: имя, телефон (+7 маска), адрес.
При отправке → показать "Заказ принят! Номер: #123".
Валидация обязательных полей.
```

**Зачем это нужно?**

- ✅ Можно быстро повторить промпт для похожей задачи
- ✅ Можно поделиться с другими (коллега, клиент)
- ✅ Можно вернуться через месяц и вспомнить, как всё делали
- ✅ Можно использовать как шаблоны для новых проектов

Это как рецепты в кулинарии. Один раз записали — потом готовите по рецепту, не изобретая заново.

### Часть 8: Правила для ИИ — файл .clinerules

Когда используете Cline (расширение VS Code), можно создать файл `.clinerules`.

Это файл с инструкциями, которые ИИ будет автоматически учитывать.

**Пример .clinerules:**

```
# Правила для разработки проекта "Доставка пиццы"

## Стиль кода:
- Использовать vanilla JavaScript (без фреймворков)
- CSS писать в одном файле style.css
- Комментарии на русском языке

## Дизайн:
- Цветовая схема: красный (#E63946) + белый
- Шрифты крупные, читаемые (минимум 16px)
- Кнопки с закруглёнными углами (border-radius: 8px)

## Telegram WebApp:
- Всегда подключать скрипт telegram-web-app.js
- Использовать Telegram.WebApp.ready() при инициализации
- Адаптировать под тему (dark/light mode)

## Структура:
- HTML файлы в папке /src
- Изображения в папке /images
- Все стили в одном файле style.css
```

**Зачем это нужно?**
- ✅ ИИ будет следовать вашим правилам автоматически
- ✅ Не нужно каждый раз повторять «используй красный цвет»
- ✅ Код получается единообразным
- ✅ Экономит время на объяснениях

Это как инструкция для помощника: «Когда готовишь кофе, всегда добавляй 2 ложки сахара, а не 1». Один раз сказали — помощник запомнил и всегда так делает.

## Шаг 2: Логика проекта — интерфейс → логика → данные (10 мин)

### Часть 1: Три слоя приложения

Представьте приложение как торт из трёх коржей 🍰. Каждый корж — отдельный слой. Все вместе = работающее приложение.

#### Слой 1: ИНТЕРФЕЙС (UI — User Interface)

Это то, что видит пользователь. Всё визуальное:
- Кнопки
- Текст
- Картинки
- Формы
- Цвета

**Для нашего приложения доставки пиццы:**
- Каталог пицц (6 карточек с картинками, названиями, ценами)
- Кнопки «Добавить в корзину»
- Страница корзины (список выбранных пицц)
- Форма заказа (поля: имя, телефон, адрес)
- Экран «Заказ принят!»

Это первый слой. Оболочка. Пользователь видит красивую картинку, но пока ничего не работает.

#### Слой 2: ЛОГИКА (Business Logic)

Это то, что происходит, когда пользователь что-то делает. Поведение приложения:
- Что случится при клике на кнопку?
- Как считается итоговая сумма?
- Что происходит при отправке формы?

**Для нашего приложения:**
- Кликнул «Добавить в корзину» → пицца добавилась в список
- Кликнул «+/−» в корзине → количество изменилось, сумма пересчиталась
- Кликнул «Удалить» → товар исчез из корзины
- Заполнил форму → проверка полей → если ОК → показать «Заказ принят!»

Это второй слой. Мозг приложения. Благодаря логике приложение становится живым.

#### Слой 3: ДАННЫЕ (Data)

Это то, где хранится информация. База данных или хранилище:
- Список товаров
- Содержимое корзины
- История заказов

**Для нашего приложения:**
- Список пицц (названия, цены, описания) — хранится в коде или JSON
- Корзина пользователя (какие пиццы добавлены) — хранится в LocalStorage
- Заказы (кто, что, когда заказал) — можно хранить в Supabase (позже)

Это третий слой. Память приложения. Без данных при перезагрузке страницы всё забывается.

### Часть 2: Почему именно такой порядок?

Важный вопрос: в каком порядке строить эти слои?

**Правильный порядок:**

1. **Интерфейс** → 2. **Логика** → 3. **Данные**

**Почему именно так?**

#### Сначала интерфейс:

Потому что так вы сразу видите результат.

Создали каталог пицц → открыли в браузере → красиво! Это мотивирует. Вы понимаете, что движетесь вперёд. Можно показать друзьям, клиенту — получить обратную связь.

**Если начать с базы данных:**
Вы две недели настраиваете таблицы, связи... А визуально ничего не видно. Скучно. Непонятно, зачем это. Поэтому сначала — красивая картинка!

#### Потом логика:

После того, как интерфейс готов, добавляем поведение. Кнопки начинают работать. Что-то происходит при кликах. Приложение оживает! Это второй уровень мотивации — вы видите, что ваше творение реагирует на действия.

#### В последнюю очередь — данные:

Когда интерфейс красивый и логика работает — добавляем сохранение. Теперь корзина не обнуляется при перезагрузке. Заказы записываются в базу. Это финальный штрих. Приложение становится полноценным.

> **Правило:** Двигайтесь от видимого к невидимому. Сначала то, что видно глазами → потом логика → потом хранение.

### Часть 3: Схема работы нашего приложения

Давайте разберём конкретно, как работает приложение доставки пиццы. Пройдём путь пользователя от начала до конца.

#### Шаг 1: Пользователь открывает приложение

**Интерфейс:**
- Видит каталог из 6 пицц
- Каждая пицца: картинка, название, описание, цена, кнопка «В корзину»

**Логика:**
- Пока нет. Просто картинка.

**Данные:**
- Список пицц захардкожен в коде (массив JavaScript)

---

#### Шаг 2: Пользователь кликает «Добавить в корзину»

**Интерфейс:**
- Кнопка меняет вид (например, зелёная галочка)
- Счётчик корзины увеличивается (плавающая кнопка «Корзина (1)»)

**Логика:**
- JavaScript обрабатывает клик
- Добавляет пиццу в массив `cart = []`
- Обновляет счётчик на кнопке «Корзина»

**Данные:**
- Массив cart хранится в памяти (пока)
- Позже сохраним в LocalStorage

---

#### Шаг 3: Пользователь открывает корзину

**Интерфейс:**
- Открывается страница cart.html
- Список добавленных пицц
- У каждой: название, цена, кнопки +/−, кнопка «Удалить»
- Внизу: итоговая сумма, кнопка «Оформить заказ»

**Логика:**
- JavaScript читает массив cart
- Генерирует HTML для каждой пиццы в списке
- Считает итоговую сумму (цена × количество)

**Данные:**
- Массив cart из LocalStorage
- Если страница перезагружена → данные остаются

---

#### Шаг 4: Пользователь изменяет количество

**Интерфейс:**
- Кликает «+» → число увеличивается
- Кликает «−» → число уменьшается
- Итоговая сумма пересчитывается автоматически

**Логика:**
- JavaScript обновляет количество в массиве cart
- Пересчитывает сумму
- Обновляет интерфейс (число и сумма меняются)

**Данные:**
- Обновлённый массив сохраняется в LocalStorage

---

#### Шаг 5: Пользователь оформляет заказ

**Интерфейс:**
- Открывается страница order.html
- Форма с полями: имя, телефон, адрес
- Кнопка «Оформить заказ»

**Логика:**
- JavaScript проверяет заполнение полей (валидация)
- Если пусто → показать «Заполните все поля»
- Если заполнено → сгенерировать номер заказа (случайный)

> «Send me the title for your Web App.»

Введите название (это увидит пользователь):
```
Доставка пиццы
```

Или:
```
Pizza Delivery
```

На ваш выбор. Нажмите Enter.

#### Шаг 5: Ввести описание

BotFather спросит:
> «Send me a description for the Web App.»

Введите описание:
```
Закажите вкусную пиццу прямо в Telegram!
```

Нажмите Enter.

#### Шаг 6: Вставить ссылку на приложение

BotFather спросит:
> «Send me the Web App URL. It should be a valid URL and start with https://»

Вставьте ссылку с Vercel, которую скопировали:
```
https://pizza-delivery-ваш-username.vercel.app
```

Нажмите Enter.

#### Шаг 7: Загрузить иконку

BotFather спросит:
> «Send me a 640x360 photo for the Web App.»

Нужна картинка размером 640x360 пикселей.

**Если есть готовая картинка:**
- Отправьте её

**Если нет:**
- Можно пропустить, написав: `/skip`
- Или быстро создать через ИИ

Для MVP можно `/skip` — иконку добавите позже.

#### Шаг 8: Завершение

BotFather ответит:
> «Done! Your Web App is ready. You can access it via this link: t.me/ваш_бот/название_app»

Готово! Mini App подключён к боту!

#### Шаг 9: Тестирование — открыть Mini App в Telegram

Теперь самое интересное — откроем приложение в Telegram!

**На телефоне (рекомендую):**
1. Откройте своего бота в Telegram (найдите его через поиск)
2. Внизу появилась кнопка с названием вашего приложения («Доставка пиццы»)
3. Нажмите на неё

**Что произойдёт:**
Внутри Telegram откроется ваше приложение — каталог пицц!

Вы увидите:
- ☑ Все 6 пицц
- ☑ Кнопки работают
- ☑ Дизайн адаптировался под Telegram

Это ваше приложение работает прямо в Telegram! Вы сделали это! 🎉🍕
